# 新的记录格式示例

## 概述

从文件级属性改为**记录级属性**，每条记录都有自己独立的类型、标题、重要程度、项目和状态。

## 文件结构

### 文件级 Frontmatter（简化）

```markdown
---
日期: 2026-02-05
tags: [#会议记录, #年度复盘, #决策, #技术]
更新时间: 2026-02-05 15:30
---
```

文件级的 Frontmatter 只包含：
- **日期**: 文件对应的日期
- **tags**: 所有记录的标签合并（去重）
- **更新时间**: 最后更新时间

### 记录级属性（每条记录独立）

```markdown
### 13:56 - 年度复盘会议

| 属性 | 值 |
|------|-----|
| 类型 | 会议记录 |
| 重要程度 | 4⭐ |
| 项目 | 没有值 |
| 状态 | 已完成 |

今天参加了年度复盘会议，讨论了去年的成果和今年的规划。

---

### 15:30 - 决策 - 采用微服务架构

| 属性 | 值 |
|------|-----|
| 类型 | 决策 |
| 重要程度 | 5⭐ |
| 项目 | 电商平台重构 |
| 状态 | 进行中 |

经过团队讨论，决定采用微服务架构替代现有的单体架构。
主要考虑因素：
1. 可扩展性更好
2. 团队可以独立部署
3. 技术栈可以更灵活

---
```

## 优势对比

### 旧格式（文件级属性）

```markdown
---
日期: 2026-02-05
时间: "13:56"
类型: "会议记录"
标题: "年度复盘会议"
tags: [#会议记录, #年度复盘]
重要程度: 4
项目: ""
状态: "已完成"
更新时间: 2026-02-05 13:56
---

## 13:56 - 年度复盘会议

今天参加了年度复盘会议。

---

## 15:30 - 决策 - 采用微服务架构

经过团队讨论，决定采用微服务架构...
```

**问题**:
- 整个文件只能有一个类型（会议记录）
- 整个文件只能有一个重要程度（4⭐）
- 第二条记录（决策）没有自己的属性

### 新格式（记录级属性）

```markdown
---
日期: 2026-02-05
tags: [#会议记录, #年度复盘, #决策, #技术]
更新时间: 2026-02-05 15:30
---

### 13:56 - 年度复盘会议

| 属性 | 值 |
|------|-----|
| 类型 | 会议记录 |
| 重要程度 | 4⭐ |
| 项目 | 没有值 |
| 状态 | 已完成 |

今天参加了年度复盘会议。

---

### 15:30 - 决策 - 采用微服务架构

| 属性 | 值 |
|------|-----|
| 类型 | 决策 |
| 重要程度 | 5⭐ |
| 项目 | 电商平台重构 |
| 状态 | 进行中 |

经过团队讨论，决定采用微服务架构...
```

**优势**:
- ✅ 每条记录有自己的类型（会议记录 vs 决策）
- ✅ 每条记录有自己的重要程度（4⭐ vs 5⭐）
- ✅ 每条记录有自己的项目和状态
- ✅ 更清晰的视觉区分
- ✅ 便于单独引用某条记录（通过锚点 `#15-30-决策-微服务`）

## 支持的属性

| 属性 | 说明 | 可选值 |
|------|------|--------|
| **类型** | 记录的类型 | 决策、会议记录、学习笔记、用户偏好、每日日志 |
| **重要程度** | 重要程度评级 | 1-5 ⭐ |
| **项目** | 关联的项目名称 | 任意文本 |
| **状态** | 记录的状态 | 进行中、已完成、暂停、已归档 |

## 链接引用

新格式支持直接链接到单条记录：

```markdown
[[memory/2026-02-05.md#10-00-决策-数据库选择]]
[[memory/2026-02-05.md#14-00-会议-架构评审]]
```

在 MEMORY.md 中同步时也会使用这种精确链接：

```markdown
## 重要决策历史

- [决策 - 数据库选择]([[memory/2026-02-05.md#10-00-决策-数据库选择]]) - 2026-02-05
- [决策 - 使用 React]([[memory/2026-02-05.md#10-30-决策-React]]) - 2026-02-05
```
